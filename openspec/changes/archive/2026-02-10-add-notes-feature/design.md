## Context

既存システムにはノートを独立管理する機能がなく、ユーザーが備忘を残すための永続的な領域が不足している。ノートはユーザーにひも付く単独機能として新規追加し、基本的なCRUD APIのみを提供する。検索や権限制御は初期段階では対象外とする。

## Goals / Non-Goals

**Goals:**

- ユーザーにひも付くノートを作成・取得・更新・削除できるようにする。
- ノートの最小フィールド（タイトル、本文、作成日時、更新日時）を保持する。
- ノートをカテゴリで分類し、カテゴリのCRUDを提供する。
- 既存のFastAPI構成（models/schemas/services/endpoints）の設計に沿って実装する。

**Non-Goals:**

- 検索や高度なフィルタリング機能。
- 編集権限・公開範囲などのアクセス制御。
- 監査ログや履歴管理。

## Decisions

- **データモデル**: `notes` テーブルを新設し、`user_id` でユーザーと関連付ける。必須フィールドとして `title`、`body`、`created_at`、`updated_at` を持つ。
- **カテゴリモデル**: `note_categories` テーブルを新設し、`user_id` でユーザーと関連付ける。必須フィールドとして `name`、`created_at`、`updated_at` を持つ。ノートは `category_id` を任意で保持し、1ノートにつき1カテゴリにひも付く形にする。
- **API設計**: `/notes` に対する CRUD を提供する（作成、一覧、単体取得、更新、削除）。一覧はユーザー単位で取得できるように `user_id` を入力条件にする。
- **カテゴリAPI設計**: `/note-categories` に対する CRUD を提供する（作成、一覧、単体取得、更新、削除）。一覧はユーザー単位で取得できるように `user_id` を入力条件にする。
- **実装構成**: 既存の `models/`, `schemas/`, `services/`, `api/endpoints/` にそれぞれノート用のモジュールを追加し、ルータに登録する。
- **バリデーション**: タイトルと本文は必須。最大長などの詳細は後続の仕様策定で定義する。

## Risks / Trade-offs

- [作成者フィールドを持たないため作成者名が取得できない] → 初期要件では不要とし、必要になった時点でスキーマ拡張を検討する。
- [一覧取得が肥大化] → 初期はページング要件なしとし、必要になった時点で仕様を追加する。
- [権限制御なし] → 誤操作リスクはあるが、当初の要件に合わせてスコープ外とする。
- [カテゴリの階層化要件が不明] → 初期はフラットなカテゴリのみ対応し、必要になった時点で階層化を検討する。

## Migration Plan

- `notes` テーブル作成のマイグレーションを追加する。
- `note_categories` テーブル作成のマイグレーションを追加する。
- ロールバック用 SQL を用意し、テーブル削除で戻せるようにする。
- 既存データへの影響はないため、追加のみで完了する。
